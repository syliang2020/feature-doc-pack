# API Contract

用于生成或更新 `API-CONTRACT.md`。接口文档必须达到前后端可并行开发的粒度。

## 每个接口必须包含

开发前设计或前后端并行开发场景下，每个接口必须达到可实现、可联调、可验收粒度。信息暂时无法确认时，不要省略，写入 `待确认事项`。

1. 接口目的
2. 请求方式和路径
3. 认证和权限规则
4. 请求字段表
5. 响应字段表或响应示例
6. 错误场景和提示语
7. 副作用
8. 分页、排序、筛选、去重规则
9. 与业务规则文件的对应关系
10. 接口编号，例如 `API-001`
11. 关联任务编号和测试用例编号

上述编号必须是可点击链接，不能只写纯文本。

## 对象定义和接口视图

接口文档可以定义通用对象，但不能只引用对象名来替代接口契约。

必须同时提供：

- 对象索引：每个 DTO、VO、分页对象、通用响应对象都要能点击跳转到定义。
- 对象定义：说明对象的全部字段含义和通用来源。
- 接口级字段视图：每个接口单独说明本接口实际使用或返回哪些字段、是否必填、是否只读、是否忽略、默认值和错误提示。
- 就近字段明细：接口正文里必须直接展开本接口请求和响应字段表。对象名链接只能作为类型引用或补充说明，不能让读者必须跳到“通用对象定义”才能知道字段。
- 可点击格式：对象链接不能放在反引号代码片段里。不要写 `` `List<[SomeVO](#schema-somevo)>` ``；改写为 `List，元素类型：` 后接普通 Markdown 对象链接，链接目标使用 `API-CONTRACT.md#schema-somevo`。

通用对象的 `必填` 只能写“按接口”，不能把某个接口的必填规则写成全局规则。比如同一个 `QueryVO` 在详情接口中 `id` 必填，在列表接口中 `id` 不使用；同一个 `SaveVO` 在新增接口中 `id` 不传，在编辑接口中 `id` 必填。

如果一个对象在不同接口中的返回字段不同，必须拆出接口视图，例如 `ResourceListItemView`、`ResourceDetailView`，或者在每个接口响应字段表中明确字段是否返回。不要只写 `响应业务数据：ResourceVO`。

禁止写法：

- `本接口响应字段：<响应包裹>.<业务数据字段> 为 List<SomeVO>，字段见对象定义。`
- `请求字段：SomeSaveVO。`
- `响应业务数据：<分页对象><SomeVO>。`

推荐写法：

1. 先按项目实际返回结构写响应形态，例如：`响应包裹：项目实际统一返回对象`、`业务数据字段：项目实际字段名`、`业务数据类型：List`，元素类型链接到 `API-CONTRACT.md#schema-somevo`。如果项目没有统一响应包裹，直接写响应 body 结构。
2. 紧接着写“本接口响应 item 字段”表，列出字段、类型、是否返回、说明、空值规则。
3. 分页接口必须按项目实际分页结构展开分页壳字段和列表 item 字段；不要预设分页字段一定叫 `data`、`records`、`list` 或其它固定名称。
4. 树、附件、分类等嵌套对象至少展开当前接口会用到的一层字段；更深层字段可以继续用链接补充，但关键联调字段不能只放在对象定义区。
5. 类型字段中需要链接对象时，推荐写 `List，元素类型：` 后接普通 Markdown 对象链接，或写 `对象：` 后接普通 Markdown 对象链接；不要把整段类型写成代码。

## 请求字段表

请求字段表必须写清：

- 字段名
- 类型
- 是否必填
- 含义
- 取值来源
- 传 ID 还是名称
- 默认值
- 校验规则
- 落库字段

不要只写字段名列表。如果关键字段结构缺失，应补 `Open Questions`，说明哪些信息阻塞前后端并行开发。

复杂字段必须展开结构，例如分组、标签、附件、树节点、分页查询条件、批量操作数组。只写 `items` 或 `attachments` 这类字段名是不合格的。

请求字段表必须以“本接口”为单位，不得只写“请求字段：某某 VO”。即使请求体类型复用同一个 VO，也要列出本接口允许前端传的字段、必填条件和未列字段处理方式。

## 兼容字段

当接口存在新旧字段兼容时，必须把兼容关系写成明确规则，不能只写“兼容”两个字。

至少写清：

- 新契约字段：新前端或新调用方必须传哪个字段。
- 旧兼容字段：哪个字段只为历史调用保留，新前端是否禁止主动传。
- 优先级：两个字段同时存在时以后端哪个字段为准，另一个字段如何处理。
- 兜底范围：旧字段在哪些资源类型、模式、接口中可兜底，哪些模式不能兜底。
- 空值规则：新契约字段为空、空数组、缺失时分别如何处理。
- 错误提示：不能兜底时返回什么业务错误。

抽象示例：`newItems` 是新契约必填字段，`legacyItemIds` 只兼容旧请求；两者同时存在时以 `newItems` 为准。若某个业务模式需要额外的层级、分组、标签或附件绑定信息，不能用只含 ID 的旧字段兜底。

## 响应和错误

响应部分必须包含：

- 成功响应示例
- 列表、分页和详情结构
- 空数据结构
- 错误码或错误消息
- 权限失败、参数错误、业务校验失败的提示语

响应对象也要按接口展开字段。对象名可以作为类型链接，但不能代替字段表。详情接口、列表接口、保存接口、删除接口的响应字段通常不同，必须分别写清。

开发前文档中，保存、编辑、删除、导入、发布等写接口必须列出成功响应和至少一类业务失败响应。列表接口必须列出分页结构、空数据结构和排序规则。

不得用“字段见对象定义”“见通用对象”“同上”代替接口级响应字段表。即使字段和通用对象完全一致，也要在接口附近提供一份紧凑字段表，方便前端、后端、测试直接按当前接口开发和验收。
响应包裹、业务数据字段和分页字段必须以当前项目为准，不能把某个项目的统一响应对象、业务数据字段或分页字段名写成全局默认。
不得把对象链接放入反引号代码片段中；质量检查时应把“看起来是链接但无法点击”的写法视为不合格。

## 接口拆分和分工

多人后端开发时，应按接口领域或数据边界拆分，并在 `TASK-BOARD.md` 中引用接口编号。常见拆分方式：

- 查询类接口：选项、树、列表、详情
- 写入类接口：新增、编辑、删除、导入、发布
- 公共能力：权限、缓存、字典、文件或附件绑定

接口文档本身不要写“后端负责接口”这种粗粒度描述，应让不同后端同事能拿到自己负责的 `API-xxx` 直接实现。

## 联调高风险点

特别要明确这些问题：

- 字段传 ID 还是名称。
- 多个字段表达同一含义时谁优先。
- 列表筛选字段和保存字段是否一致。
- 新旧字段同时存在时以后端哪个字段为准。
- 旧字段是否只是兼容兜底，是否禁止新前端主动传。
- 附件或元数据字段的完整结构、绑定键和回填规则。
- 当前阶段不做的接口不得写成当前交付接口。

## 副作用

如果接口会写数据，必须说明：

- 新增、更新、删除哪些表。
- 是否重建关联表。
- 是否会反查字典或补充名称。
- 是否会校验当前登录人、身份、所属关系。
- 是否影响其他模块已有数据。

每个接口必须包含：

如果接口还没有对应规则或测试，写入“覆盖缺口”，不要假装完整。

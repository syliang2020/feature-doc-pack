# Document Levels

用来判断用户选择 2、3、4 时应生成或更新哪些文件。

## 最小文档

适用于：

- 单模块小迭代
- 小范围接口调整
- 1 到 2 人可完成的需求

生成或更新：

- `PLAN.md`
- `API-CONTRACT.md`，仅在涉及接口时
- `BUSINESS-RULES.md`，仅在业务规则复杂、需要共享细则时

最小文档通常不强制单独写进度视图，但如果涉及多人协作，应在 `PLAN.md` 或 `TASK-BOARD.md` 中至少标明负责人和状态。

## 标准文档包

适用于：

- 2 人及以上协作
- 前后端联调
- 跨模块改动
- 中等复杂度需求

生成或更新：

- `PLAN.md`
- `BUSINESS-RULES.md`
- `API-CONTRACT.md`
- `TASK-BOARD.md`
- `TEST-CASES.md`

其中 `TASK-BOARD.md` 必须包含进度视图，至少能看出负责人或待分配、状态、依赖、阻塞和完成标准。

## 完整文档包

适用于：

- 跨仓库协作
- 多团队参与
- 风险较高
- 需要持续联调和问题追踪
- 提测或发布链路复杂

生成或更新：

- `PLAN.md`
- `BUSINESS-RULES.md`
- `API-CONTRACT.md`
- `TASK-BOARD.md`
- `TEST-CASES.md`
- `CHANGELOG.md`
- `ISSUES.md`
- `RELEASE.md`
- `PROGRESS.md`，仅在多人长期协作、用户要求单独看进度、或任务板过大时生成

## 文件职责

- `PLAN.md`：说明本次改动做什么、不做什么、谁负责什么、里程碑、风险点、验收标准。只放业务规则摘要。
- `BUSINESS-RULES.md`：说明详细业务规则，包括状态流转、字段级规则、复杂校验、模式差异、限制条件、统计口径、异常提示、边界场景；不写 DDL、索引、迁移、回填或回滚 SQL。
- `API-CONTRACT.md`：说明接口、字段、错误码、枚举值、请求响应结构和必要数据映射；不写数据库迁移脚本。
- `DATA-MODEL.md`：说明表结构影响、字段或索引变更、迁移 SQL、数据回填、兼容策略、回滚方案和数据库验证点。
- `TASK-BOARD.md`：拆分角色或负责人任务，避免多人重复开发或边界冲突，并记录任务状态、依赖和阻塞。
- `TEST-CASES.md`：记录联调点、验收点、边界和异常场景，并映射业务规则和接口编号。
- `CHANGELOG.md`：记录接口、字段、规则变化，仅在确实发生变化时更新。
- `ISSUES.md`：记录联调或开发中的问题。
- `RELEASE.md`：用于提测、上线、回滚和发布检查，仅在需要交付发布时使用。
- `PROGRESS.md`：独立记录总体进度、里程碑、阻塞、风险和下一步动作，仅在进度信息会让任务板过重时使用。

## 可选附加文档

可选附加文档不属于默认文档包。只有用户明确确认、项目已有同类文档需要更新，或需求一开始就说明需要数据库/发布专项文档时，才生成或更新。

- `DATA-MODEL.md`：说明表结构影响、字段或索引变更、迁移 SQL、数据回填、兼容策略、回滚方案和数据库验证点。

如果在文档评审或后续优化时才发现需要 `DATA-MODEL.md`，不要直接创建。先在现有文档的 `待确认事项` 中说明“是否新增数据库变更文档”，并询问用户确认。

## 追踪编号

开发前、开发中、标准文档包和完整文档包应优先使用稳定编号：

- 业务规则：`BR-001`
- 接口：`API-001`
- 数据模型或迁移：`DB-001`，仅在已确认存在 `DATA-MODEL.md` 或项目迁移文档时使用
- 任务：`TASK-FE-001`、`TASK-BE-001`、`TASK-QA-001`
- 测试用例：`TC-001`

编号用于串联“业务规则 -> 接口 -> 任务 -> 测试”。不要求为每一句话编号，但关键规则和关键接口必须可追踪。
当用户确认维护数据库变更文档时，`DB-xxx` 也应被任务和测试用例引用，并能跳转到 `DATA-MODEL.md` 或项目迁移文档中的对应条目。

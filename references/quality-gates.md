# Quality Gates

生成或更新协作文档时，用这些规则防止文档“看起来完整但不能开工”。

## 范围一致性

生成或更新文档前，先锁定本次改动的范围边界：

- 当前文档阶段：开发前设计、开发中协作、开发后恢复/复盘、提测/发布
- 当前阶段必须做什么
- 当前阶段明确不做什么
- 哪些内容属于后续阶段
- 哪些内容需要用户确认后才能写入当前范围

`PLAN.md` 是范围基准。其他文件不得把 `PLAN.md` 中明确排除的功能写成本期开发内容。

如果 `BUSINESS-RULES.md`、`API-CONTRACT.md`、`TASK-BOARD.md` 或 `TEST-CASES.md` 需要提到后续阶段能力，必须标记为“后续阶段 / 不在本期实现”。

如果发现范围冲突：

- 不要自行猜测。
- 在文档中新增 `Open Questions` 或 `待确认事项`。
- 写清楚冲突来源。
- 只在用户确认后再把冲突内容并入本期范围。

## 文档阶段质量门槛

开发前设计文档必须让团队能直接开工：

- `PLAN.md` 写目标、范围、里程碑、分工方式和验收标准，不写成“恢复说明”。
- `API-CONTRACT.md` 写完整接口契约，不用字段名列表代替请求响应结构。
- `API-CONTRACT.md` 的每个接口必须在接口正文附近展开请求/响应字段明细，不能让读者依赖跳转到通用对象定义才能知道字段。
- `BUSINESS-RULES.md` 写页面、字段、模式、操作和异常规则。
- `TASK-BOARD.md` 写到多人可分工的粒度，并包含进度字段。
- `TEST-CASES.md` 从业务规则反推测试用例。
- 团队人数或负责人未确认时，`TASK-BOARD.md` 仍然必须有可认领的任务包边界；不能只写“待确认”。

开发后恢复/复盘文档可以写“恢复说明”“已回滚”“当前仍保留”，但必须明确这不是开发前开工基线。若用户要求按未开发系统评估或生成文档，不得沿用开发后语气。

## PLAN.md 质量门槛

`PLAN.md` 必须作为协作范围的单一基准，并包含：

- 本次目标
- 当前阶段范围
- 明确不做内容
- 后续阶段内容
- 验收标准
- 风险点
- 待确认事项

开发前 `PLAN.md` 还必须说明：

- 系统或功能要解决什么问题
- 涉及哪些页面、接口、角色和数据对象
- 团队如何分工，未知负责人写“待分配”
- 哪些内容阻塞开工，进入 `待确认事项`

`不做内容` 是硬边界。后续文档如果引用这些能力，必须标记为后续阶段，不能让开发者误以为本期要实现。

## Legacy repository 规则

对于老项目：

- 不要求为整个系统从 0 到 1 补全历史文档。
- 只为当前需求相关的功能切片创建或更新文档。
- 优先复用现有文档。
- 没有现成文档时，只按用户本次选择的级别补最小必要内容。

建议路径：`feature/<feature-name>/`。

## 写作底线

- 不要只建空文件。
- 不要写与当前改动无关的内容。
- 不要假设历史系统都已文档化。
- 不要为了“完整”而过度设计。
- 全局 skill 或模板示例不得夹带某个具体项目的字段名、业务模式、枚举值、表名、接口路径或响应对象；需要举例时使用中性占位。
- 不要静默新增超出用户所选文档级别的文件；需要新增附加文档时先说明原因并取得确认。
- 不要把未确认的猜测写成已确认规则。
- 不要把用户举例、反问或质疑中的团队人数、负责人、排期、优先级写成已确认事实。
- 文档要能直接服务本次开发、联调、提测。
- 不要把数据库 DDL、索引、迁移 SQL、回填 SQL 或回滚 SQL 放进 `BUSINESS-RULES.md`。

## 数据库变更质量门槛

涉及数据库结构、索引、迁移、回填或回滚时，必须能回答：

- 变更哪张表、哪个字段或哪个索引。
- 变更原因对应哪条业务规则或接口。
- 字段类型、长度、默认值、是否允许空、注释、枚举或格式约束。
- 是否需要数据回填、兼容旧数据、清理脏数据。
- 是否影响查询性能，是否需要索引或执行计划确认。
- 上线顺序、回滚方式和验证 SQL。
- 对应的后端任务、测试用例和发布检查。

这些内容不要写入 `BUSINESS-RULES.md`。优先写入项目已有迁移文档、迁移脚本或 `RELEASE.md`；如果需要新增 `DATA-MODEL.md`，必须先取得用户确认。`BUSINESS-RULES.md` 只引用字段业务口径。

## 进度和追踪质量门槛

标准文档包和完整文档包必须能回答：

- 哪些任务未开始、开发中、待联调、联调中、待测试、已完成或阻塞。
- 每个任务的负责人或待分配角色是谁。
- 每个任务依赖哪些业务规则、接口或其它任务。
- 关键业务规则是否都有接口、任务和测试覆盖。
- 所有关联编号是否能点击跳转到对应定义或详情。
- Markdown 链接是否没有被包在反引号代码片段中，避免渲染后无法点击。

建议编号：

- 业务规则：`BR-001`
- 接口契约：`API-001`
- 任务：`TASK-FE-001`、`TASK-BE-001`、`TASK-QA-001`
- 测试用例：`TC-001`

如果某条规则无法映射到接口、任务或测试，必须写入覆盖缺口或待确认事项。

追踪链接规范：

- `BR-001` 指向 `BUSINESS-RULES.md#br-001`
- `API-001` 指向 `API-CONTRACT.md#api-001`
- `DB-001` 指向 `DATA-MODEL.md#db-001` 或项目已确认的迁移文档锚点
- `TASK-BE-001` 指向 `TASK-BOARD.md#task-be-001`
- `TC-001` 指向 `TEST-CASES.md#tc-001`
- 目标位置必须存在锚点。
- 不使用编号范围代替链接，多个编号逐个写出。

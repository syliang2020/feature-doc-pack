# Data Model

用于用户已确认新增或项目已存在 `DATA-MODEL.md` 时，生成或更新数据库变更文档；也用于判断数据库结构、索引、迁移、回填和回滚说明是否足够让后端、DBA、测试和发布同事协作。

`DATA-MODEL.md` 是可选附加文档，不是默认标准文档包成员。不能因为 `BUSINESS-RULES.md` 不适合放 DDL，就静默新增本文件。

## 何时维护 DATA-MODEL.md

满足以下任一情况时，应询问用户是否生成或更新 `DATA-MODEL.md`，或优先复用项目已有数据库迁移文档：

- 新增、删除或修改数据库字段
- 新增、删除或修改索引
- 字段类型、长度、默认值、是否为空、注释或枚举含义变化
- 需要数据回填、数据清理或旧数据兼容
- 查询口径变化可能影响性能
- 上线或回滚需要执行数据库脚本

如果只是描述字段业务含义，不涉及数据库结构或迁移脚本，可以只在 `BUSINESS-RULES.md` 写业务口径。

如果用户尚未确认新增 `DATA-MODEL.md`，先在 `PLAN.md`、`TASK-BOARD.md` 或 `待确认事项` 中记录数据库变更需要承接的位置，不要直接创建文件。

## Rule Placement

- `BUSINESS-RULES.md`：写字段业务含义、写入时机、筛选口径、边界规则。
- `API-CONTRACT.md`：写接口入参、出参、字段语义和必要数据映射。
- `DATA-MODEL.md`：写表结构影响、DDL、索引、回填、兼容、回滚和验证 SQL。
- `TASK-BOARD.md`：写数据库变更由哪个后端任务负责。
- `TEST-CASES.md`：写迁移、兼容、筛选和回归测试。
- `RELEASE.md`：写上线顺序、执行窗口、回滚检查和发布风险。

不要在 `BUSINESS-RULES.md` 中写 `ALTER TABLE`、`CREATE INDEX`、迁移 SQL、回填 SQL 或回滚 SQL。

## DATA-MODEL.md 推荐结构

1. 适用范围
2. 数据库变更编号，例如 `DB-001`
3. 表结构变更
4. 字段和索引说明
5. 迁移脚本
6. 数据回填或旧数据兼容
7. 查询性能和索引风险
8. 上线顺序
9. 回滚方案
10. 验证 SQL
11. 与业务规则、接口、任务、测试的对应关系

## 数据库变更质量门槛

每个数据库变更条目至少说明：

- 变更编号，例如 `<a id="db-001"></a>[DB-001](DATA-MODEL.md#db-001)`。
- 变更对象：库、表、字段、索引。
- 变更原因：关联业务规则、接口或性能问题。
- 字段定义：类型、长度、默认值、是否为空、注释、格式或枚举。
- 兼容策略：旧数据如何处理，是否需要回填，空值如何兼容。
- 性能影响：是否影响查询条件、排序、关联、索引命中。
- 执行方式：开发环境、测试环境、生产环境的执行脚本或迁移文件。
- 回滚方式：如何撤销结构或数据变更，是否会丢数据。
- 验证方式：上线前后用哪些 SQL 或系统操作确认变更成功。

关联业务规则、接口、任务、测试用例等编号必须是可点击链接。

## 常见遗漏

- 只写业务需要字段，但没有字段类型、长度和是否为空。
- 写了 `ALTER TABLE`，但没有写旧数据兼容和回滚方案。
- 查询条件变化却没有说明索引或性能风险。
- 只写开发库脚本，没有说明测试和生产如何执行。
- 数据库变更没有映射到后端任务、测试用例或发布检查。

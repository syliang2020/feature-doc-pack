# Feature Doc Pack

你是一个协作文档生成与维护工作流。目标是在真正需要时，为当前 feature 切片生成最小必要的协作文档。

核心原则：

- 先询问，再生成；用户拒绝后，本次改动不生成协作文档。
- 只服务当前改动，不补全整个老项目历史文档。
- 文档清单必须稳定；按用户选择的文档级别生成，不要在后续优化中静默新增超出清单的文档。
- 本 skill 是全局规则，示例只能使用抽象字段名、占位符或中性场景；不要写入某个项目的字段名、业务类型、枚举值、表名、接口名或返回对象。
- 先判断文档阶段：开发前设计、开发中协作、开发后恢复/复盘、提测/发布。
- 开发前文档必须以“其他开发者能直接开工”为目标，不能写成开发后的恢复说明。
- `PLAN.md` 是范围基准，其他文档不能突破它的“不做内容”。
- 复杂业务规则进入 `BUSINESS-RULES.md`，不要全部堆到 `PLAN.md`。
- 数据库结构、迁移 SQL、索引、数据回填和回滚步骤不属于 `BUSINESS-RULES.md`；业务规则只写数据口径、字段语义和业务触发条件。
- `API-CONTRACT.md` 必须让开发者在当前接口段落附近看到请求/响应字段明细；对象定义链接只能补充，不能代替接口字段表。
- Markdown 链接不能放在反引号代码片段里；需要展示类型时，写成 `List，元素类型：` 后接普通 Markdown 对象链接，链接目标使用 `API-CONTRACT.md#schema-xxxvo`，保证对象名可点击。
- 标准及完整文档包必须能看出进度；业务规则、接口、任务和测试应建立可点击的追踪编号。
- 与 superpowers 协作时，本 skill 只负责协作文档判断和质量门槛；计划、调试、TDD、验证仍按对应 superpowers skill 推进。

## Skill Boundary

本 skill 负责判断和生成当前功能切片的协作文档，不替代后端设计审查。

当任务涉及后端数据模型质量、字段语义、命名/分层、校验分层、save/update 复用、事务边界、外部调用或历史数据兼容时，应先使用 `backend-feature-design-review` 做设计审查。设计方向确认后，如仍需要协作文档，再使用本 skill。

## Trigger

当任务可能涉及以下任一情况时，先判断是否建议生成协作文档：

- 多人协作、前后端联调、跨模块或跨仓库改动
- 新增接口、接口字段变更、字段含义变更
- 数据库字段、索引、迁移脚本、数据回填或存储口径变化
- 多种业务模式、复杂状态流转、复杂校验或复杂统计
- 删除、编辑、发布、导入、附件、回填等存在限制条件
- 多个页面或前后端共享同一批业务细则
- 需要跟踪开发进度、阻塞、联调状态或规则到测试的覆盖情况

小 bug、文案修改、样式微调、单点逻辑修复，通常不要主动建议生成文档。

## Ask First

命中触发场景时，不要直接创建文件。先询问用户一次：

```text
这次改动涉及多人协作 / 联调 / 接口变更 / 复杂业务规则，建议为本次改动生成协作文档。
可选：
1. 不生成文档，直接开发：适合小改动，或你已经明确需求边界
2. 生成最小文档：只补本次改动最核心的计划、接口和必要规则
3. 生成标准文档包：适合多人协作或联调，补计划、规则、接口、任务拆分和测试点
4. 生成完整文档包：适合跨仓库、高风险、提测或发布链路复杂的需求
如果你不确定，通常优先选 3。请选择一种。
如果是多人协作，请同时说明前端、后端、测试的大致人数；未说明时我会按模块或角色拆分，并把团队配置、负责人、人员分工写成待确认或待分配。
用户用来追问、举例或反问的人数、角色、排期、负责人，不等于本次已确认事实。除非用户明确确认，否则不得写成计划基线、任务分工或进度视图。
```

用户同意后，按用户选择的级别生成或更新文档，不越级扩张。用户拒绝后，本次改动不生成协作文档，也不要在同一需求上下文中再次主动提出。

## Document Stage

生成或更新文档前，必须先判断文档阶段：

- 开发前设计：系统或功能尚未开发，文档目标是可开工、可分工、可联调、可验收。
- 开发中协作：代码已部分实现，文档目标是同步范围、进度、阻塞和联调状态。
- 开发后恢复/复盘：功能已实现，文档目标是恢复规则、记录回归点和遗留问题。
- 提测/发布：功能准备交付，文档目标是测试覆盖、发布检查、风险和回滚。

如果用户说“假设还没开发”“团队开发”“按文档开工”，默认按开发前设计处理。开发前文档不得使用“恢复说明”“已回滚”“当前仍保留”等开发后语气，除非用户明确要求记录历史上下文。

## Doc Levels

详细级别说明见 [doc-levels.md](references/doc-levels.md)。

- 最小文档：`PLAN.md`，涉及接口时加 `API-CONTRACT.md`，业务规则复杂时加 `BUSINESS-RULES.md`。
- 标准文档包：`PLAN.md`、`BUSINESS-RULES.md`、`API-CONTRACT.md`、`TASK-BOARD.md`、`TEST-CASES.md`，其中 `TASK-BOARD.md` 必须包含进度视图。
- 完整文档包：标准文档包加 `CHANGELOG.md`、`ISSUES.md`、`RELEASE.md`；多人长期协作或用户要求看进度时，可加 `PROGRESS.md`。
- `DATA-MODEL.md` 不是默认文档包成员。只有用户确认、项目已有同类文档、或用户选择完整文档包且明确涉及数据库迁移/回填/回滚时，才生成或更新；不要为了解决内容归属问题静默新增。

## Workflow

1. 如果任务同时命中后端设计审查和协作文档场景，先完成 `backend-feature-design-review`，再决定是否生成协作文档。
2. 识别文档阶段、功能范围、角色、团队人数、仓库、接口变更、联调需求、提测/上线需求、复杂业务规则。
3. 区分已确认事实、用户举例/反问、合理推断和待确认事项；团队人数、负责人、排期、优先级没有明确确认时，只能写待确认或待分配。
4. 即使团队人数或负责人待确认，也必须先按任务包定义清楚开发边界、输入依赖、输出物、联调对象和完成标准；待确认只能影响“谁负责”，不能影响“做什么”和“做到什么程度”。
5. 如果需要文档，先询问用户并记录本次选择。
6. 生成或更新文档前，锁定当前阶段范围、明确不做内容和后续阶段内容。
7. 对开发前或多人协作文档，先建立追踪编号规则和链接规则：`BR-xxx`、`API-xxx`、`TASK-xxx`、`TC-xxx` 都必须能点击跳转到对应定义或详情。
8. 如果发现需要超出当前文档包清单的新文件，例如 `DATA-MODEL.md`、`RELEASE.md`、`PROGRESS.md`，先在现有文档的待确认事项中说明原因，并询问用户是否新增，不要直接创建。
9. 按场景加载下方 reference，避免一次性读取所有细节。
10. 生成或更新后执行质量检查：阶段一致、范围一致、接口可联调、规则可执行、任务可分工、进度可查看、测试可验收、编号可追踪、未确认事项没有被写成事实。

## Reference Loading

- 生成任意文档包时，读取 [quality-gates.md](references/quality-gates.md)。
- 涉及复杂业务规则时，读取 [business-rules.md](references/business-rules.md)。
- 涉及新增接口、接口字段或前后端联调时，读取 [api-contract.md](references/api-contract.md)。
- 涉及数据库结构、索引、迁移、回填或回滚时，先判断是否已有项目迁移脚本或用户是否确认新增 `DATA-MODEL.md`；确认后再读取 [data-model.md](references/data-model.md)。
- 需要任务拆分或测试用例时，读取 [task-board-and-tests.md](references/task-board-and-tests.md)。
- 用户要求检查已有协作文档时，读取 [review-checklist.md](references/review-checklist.md)。

## Update Policy

如果文档已存在：

- 优先更新，不重建。
- 保留有效内容。
- 明确新增、删除、变更部分。

如果文档不存在：

- 按用户确认的级别创建最小必要文件。
- 不创建空文件。
- 不写与当前改动无关的内容。

## Hard Constraints

- 未经用户同意，不生成或更新协作文档。
- 不要把“建议生成”当作“批准生成”。
- 不要让不同文档对同一功能给出互相冲突的范围结论。
- 不要用字段名列表替代接口契约。
- 不要在接口响应中只写 `字段见对象定义`、`响应业务数据为某 VO` 或 `请求字段为某 VO`；必须按项目实际响应结构在当前接口下展开字段明细。
- 不要写 `` `List<[SomeVO](#schema-somevo)>` `` 这类把链接包进代码片段的格式；这会导致链接无法点击。
- 开发前接口信息不全时，必须写入 `待确认事项`，不能直接省略。
- 标准及完整文档包不能只有任务清单，必须能看出状态、负责人或待分配、依赖和阻塞。
- `待确认` 不能替代开发边界。任务包必须让开发者在没有最终人名时，也能知道自己认领后要改哪些页面、接口、服务、字段和测试。
- 关键业务规则必须能追踪到接口、开发任务和测试用例；所有关联编号必须是可点击链接；无法追踪时写明缺口。
- 不要用 `BR-001 至 BR-005`、`TC-010 至 TC-018` 这类范围代替追踪链接。需要多个编号时逐个列出并逐个链接。
- 不要把未确认的猜测写成已确认规则。
- 不要把用户举例、反问、质疑或历史问题中的团队人数、角色数量、负责人、排期、优先级写成本次事实。
- 不要把某个项目的字段名、业务模式、固定枚举、数据库表、接口路径或响应对象写进 skill 模板示例；需要举例时使用 `primaryField`、`legacyField`、`ExampleVO`、`table_name.column_name` 这类中性占位。
- 不要把复杂业务规则全部压进 `PLAN.md`。
- 不要在 `BUSINESS-RULES.md` 中写 `ALTER TABLE`、`CREATE INDEX`、迁移 SQL、回填 SQL 或回滚 SQL。
- 不要未经确认新增 `DATA-MODEL.md`。如果当前文档包没有该文件，应先在 `PLAN.md` 或 `TASK-BOARD.md` 记录“涉及数据库变更，迁移脚本/数据模型文档待确认”，并询问用户是否新增数据库变更文档。
